# ADR 0014：组件级版本管理与标签驱动发布

状态：已接受
日期：2026-05-27

## 背景

z-fleet 由多个可独立构建、打包和发布的组件组成，包括 `agent`、`server`、`installer` 以及若干共享库。不同组件的演进节奏并不相同：有的组件需要频繁修复，有的组件主要随平台或协议演进，有的组件则只在安装或升级链路变化时发布。

如果仍以仓库统一版本作为发布依据，会带来几个问题：

- 任一组件的微小改动都可能迫使全仓库同步升版；
- 发布产物无法清晰表达“哪个组件发生了变化”；
- CI 很难准确判断应当发布哪个组件、哪个版本；
- release note、变更日志和包仓库记录容易混用多个版本语义。

z-fleet 已经采用组件化安装模型、独立 `releases/<version>` 目录和 `active-version` 启动边界，因此版本治理也需要按组件边界独立维护。

## 决策

1. **组件独立版本**

   `agent`、`server`、`installer` 分别维护自己的语义化版本号，彼此独立演进，不要求仓库整体同步版本。

2. **版本真相源以组件 tag 为准**

   发布时使用组件级 Git tag 作为版本确认依据，tag 格式统一为：

   - `agent/v<semver>`
   - `server/v<semver>`
   - `installer/v<semver>`

   其中 `<semver>` 采用语义化版本格式，例如 `1.4.0`。

3. **组件源码中保留版本文件**

   每个可发布组件在源码中维护自己的版本文件，用于声明当前目标版本，并在构建时注入到二进制和打包流程中。发布 tag 必须与对应组件的版本文件一致。

4. **CI 仅在 tag 发布时判定发布目标**

   CI 不通过分支名、提交内容或文件差异猜测发布对象。只有在推送组件 tag 时，CI 才进入发布流程，并从 tag 解析出组件名与版本号。

5. **release note 采用方案 B**

   release note 以“最近一个同组件 tag 到当前 tag 的提交范围”为基础自动生成草稿，再由维护者人工确认和补充。release note 不完全依赖纯手工维护，也不完全依赖原始 `git log` 直接输出。

6. **同一提交可对应多个组件 tag**

   若一个提交同时包含多个组件的可发布变更，可以在同一提交上打多个组件 tag。CI 分别按 tag 触发对应组件的发布流程。

## 规则

### tag 命名规则

- tag 必须以组件名作为前缀；
- tag 必须包含 `v` 前缀和语义化版本；
- 组件名必须是仓库认可的可发布组件；
- 同一组件的 tag 版本不得重复。

### 版本一致性规则

- tag 解析出的版本必须与对应组件源码中的版本文件一致；
- 若两者不一致，发布流程必须失败；
- 版本文件的修改应作为正常代码变更提交，不应由 CI 在发布时自动回写。

### release note 规则

- release note 以单组件为边界生成；
- release note 的输入范围是上一个同组件 tag 到当前 tag 的提交记录；
- release note 输出应包含本次发布的主要变更、兼容性提示和必要的制品信息；
- 最终发布前允许人工补充、重排或删改草稿内容。

### CI 规则

- 普通分支 push 和 pull request 只负责构建与测试；
- 组件 tag push 才触发对应组件发布；
- CI 必须显式识别组件名和版本号，不得隐式推导；
- 发布任务只影响对应组件，不得默认联动其他组件。

## 备选方案

- **仓库统一版本**
  - 优点：简单直观，所有组件共享一个发布号。
  - 缺点：不适合多组件独立演进，任何单组件变更都会污染全局版本。

- **仅使用一个全局 tag，如 `v1.2.3`**
  - 优点：实现最简单。
  - 缺点：无法表达具体是哪个组件发布，CI 和包仓库都需要额外猜测。

- **完全自动化 release note**
  - 优点：无需人工维护。
  - 缺点：提交信息质量会直接决定 release note 质量，可读性和可审计性不足。

- **完全手工维护 release note**
  - 优点：可控性高。
  - 缺点：维护成本高，容易遗漏，且与 tag/提交范围脱节。

## 影响

- 组件可以按自己的节奏独立发布，版本边界清晰。
- CI 可以直接通过 tag 判断发布目标，降低误发和串发风险。
- release note 可以保持较高可读性，同时保留自动化效率。
- 版本文件、tag 和发布脚本之间需要保持一致性检查，发布流程会比统一版本稍复杂。
- 后续新增组件时，需要为新组件补齐版本文件、tag 规则和发布侧配置。
