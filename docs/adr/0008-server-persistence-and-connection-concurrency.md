# ADR 0008：Server 持久化边界与连接并发模型

状态：已接受
日期：2026-05-22

## 背景

Server 当前使用 SQLite 作为默认持久化后端，控制面采用 HTTP/2 长连接。集成测试暴露出一个并发问题：测试主线程轮询读取 SQLite 文件时，Server 侧写事务持有锁，读侧遇到 `SQLITE_BUSY` 异常；异常栈展开期间未及时 `join` 的运行线程触发 `std::terminate`，最终表现为 crash。

这个问题不只是测试异常处理问题，也暴露出三个架构边界需要明确：

- SQLite 是单写者模型，即使开启 WAL，也不能把多线程写入视为无锁并发；
- 业务层当前依赖具体 `ServerDatabase` 类，尚未形成可替换持久化后端的接口边界；
- HTTP/2 Server 当前接受连接后为每个连接创建一个阻塞处理线程，连接数量、优雅停止和数据库写入竞争需要被明确约束。

项目当前目标仍然是面向个人开发者、小团队、Homelab、边缘设备和测试环境的轻量端点管理框架，不应因为一次 SQLite 锁冲突过早引入完整数据库服务、复杂异步框架或重型 RPC 运行时。

## 决策

- v0.x Server 默认继续使用 SQLite，保持单文件、低运维成本的部署模型。
- SQLite 实现必须开启 WAL，配置 `busy_timeout`，并对可恢复的 `SQLITE_BUSY` 场景做有界重试。
- Server 内部必须明确串行化写操作。读操作可以使用独立连接，但不能假设读永远不会遇到 busy；测试轮询读库也必须按可恢复错误处理。
- `ClaimNextTaskForAgent` 这类“查询并修改”的操作必须保持原子业务语义，避免多个连接或线程重复领取同一任务。
- 引入 Server 领域持久化接口，例如 `ServerStore` 或 `ControlStore`。HTTP/2 service、dispatcher 和连接处理层只能依赖领域接口，不直接依赖 `SQLiteCpp`、SQL 语句或数据库文件路径。
- 当前不以“零改动切换 PostgreSQL”为目标，但以“不修改业务编排层切换持久化实现”为目标。切换 PostgreSQL 时允许新增依赖、配置、迁移脚本和 store 实现，不应修改协议处理、控制事件编排和任务状态业务判断。
- Server 当前可保留“accept 异步 + 每连接一个阻塞线程”的 v0.x 过渡实现，但目标连接模型是少量 Asio I/O 线程、有限业务 worker pool、连接数量上限、清晰的停止流程和可靠的线程 `join`。
- SQLite 写入应收敛到 db actor（单写队列）。业务线程通过 `ServerStore` 提交领域操作，由 db actor 串行执行写事务；读操作可以保留独立只读连接，但必须继续按可恢复 busy 处理。
- Server 停止流程必须覆盖连接、业务 worker、db actor 队列和后台线程，异常路径不得遗留 joinable 线程。
- 是否切换 PostgreSQL 由部署规模、审计数据量和运维要求决定，而不是作为 v0.x 前置条件。

## 备选方案

- 只修测试，捕获 `SQLITE_BUSY` 并补 `join`：可以避免当前 crash，但不会解决 Server 写入并发边界和后续数据库替换边界，已拒绝作为完整方案。
- 全局大锁包住所有数据库操作：实现简单，能快速降低 SQLite 锁冲突，但会把读操作也串行化，后续扩展性弱。可以作为短期临时实现，但不作为长期接口设计。
- 立即切换 PostgreSQL：并发和事务能力更强，也更适合多实例 Server，但会引入额外服务依赖、配置、部署和迁移成本，不符合当前轻量 v0.x 目标。
- 立即重写为全异步 HTTP/2 Server：连接扩展性更好，但实现复杂度明显上升，且当前项目瓶颈首先在持久化并发边界，不适合作为修复 SQLite busy 问题的前置条件。
- 使用通用 SQL 抽象层隐藏 SQLite/PostgreSQL 差异：表面上可替换，但容易把 SQL 方言、事务语义和迁移差异泄漏到业务层。项目应优先使用领域持久化接口，而不是通用 SQL 门面。

## 影响

- 短期修复应优先集中在 `apps/server` 的持久化实现和集成测试退出流程，避免只在测试里吞掉异常。
- `SQLiteCpp` 应继续限制在 SQLite store 实现内部，不向 service、dispatcher、transport 或 protocol 层扩散。
- 配置模型需要为未来后端切换预留空间，例如从单一 `database_path` 演进为持久化后端类型和后端专属配置，但 v0.x 可以先保持 SQLite 默认路径。
- 集成测试不应依赖“直接读 SQLite 文件永不阻塞”的假设；如果继续直接读库，应使用 busy timeout、重试和确定性的运行线程清理。
- 后续如果引入 PostgreSQL，应新增独立 ADR 或修订本 ADR，明确部署目标、迁移策略、连接池、事务隔离级别和测试矩阵。
- 网络层在 v0.x 可以继续保持简单实现，但新增功能不得无限增加连接线程或在连接线程内执行长时间阻塞任务；大 payload 和高延迟工作应按 ADR 0007 的流量分层要求独立设计。
