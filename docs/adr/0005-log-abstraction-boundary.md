# ADR 0005：日志通过 Core 门面封装

状态：已接受
日期：2026-05-14

## 背景

v0.1 已将 `spdlog` 纳入依赖基线，但当前仓库还没有实际日志实现。项目是 `agent + server + libs` 的 monorepo，日志会横跨启动入口、网络流程、数据库、平台探测和后续任务执行。如果业务代码直接分散依赖 `spdlog`，会带来以下问题：

- 第三方依赖扩散到 `apps` 和各个 `libs`，削弱模块边界；
- 后续如果需要补充统一脱敏、上下文字段、测试 sink 或替换日志库，修改面会扩散；
- 运行日志与审计事件的职责边界容易混淆；
- 各组件可能自行拼接前缀，导致 `agent_id`、`request_id` 等字段格式不一致。

## 决策

- `spdlog` 只作为 `libs/core` 内部实现使用。
- 业务代码统一依赖 `zfleet/core/log.h` 提供的日志门面，不直接 `#include <spdlog/...>`。
- Agent 和 Server 在进程入口初始化日志系统，并从配置文件或命令行装配日志级别和输出位置。
- 日志门面应支持组件名和结构化上下文，优先承载 `agent_id`、`request_id`、`task_id`、`audit_id` 等项目内标识字段。
- 审计事件仍然通过独立审计通道入库；运行日志仅作为运行观测和排障辅助，不替代审计。

## 备选方案

- 全仓直接使用 `spdlog`：短期编码更快，但三方依赖会扩散到业务层，后续统一策略或替换成本高。
- 自建完整 `ILogger` 抽象层：可插拔性更强，但对当前项目规模过重，会引入不必要的复杂度和样板代码。
- 仅在 app 层封装，库内直接打第三方日志：边界不一致，`libs/platform` 和 `libs/protocol` 仍会暴露第三方依赖。

## 影响

- `libs/core` 负责日志初始化、级别解析、sink 组合、格式控制以及未来的统一脱敏策略。
- `apps/agent` 和 `apps/server` 可以通过配置或命令行覆盖日志级别，并共享一致的日志格式。
- 后续如需替换 `spdlog`、补充 JSON 日志、接入测试 sink 或增加额外上下文，应优先在 `libs/core` 内部演进，不改业务侧接口。
- 架构文档需要同步声明该边界，避免实现时回退为“虽然选了 `spdlog`，但各处直接调用”的模式。
