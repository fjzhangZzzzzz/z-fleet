# ADR 0013：Web 管理界面的信息架构与交互约束

状态：已接受
日期：2026-05-26

## 背景

ADR 0010 已确定 `zfleet_server` 内置 Web 管理入口的能力边界、安装包 channel、注册 token 和管理 API 范围，但没有明确约束 Web UI 的信息架构、交互密度和可见性边界。

在近期实现和评审中，团队已经形成一组更细的管理端设计决策，例如：

- Web 管理面应呈现为基础设施控制台，而不是营销页或展示页；
- 安装页应尽量减少用户输入，只暴露必要选择，不暴露后端部署细节；
- Agent 详情抽屉中的维护动作应附着在版本上下文中，而不是使用突兀的大主按钮；
- 中文界面优先，视觉密度应偏紧凑，避免为展示感牺牲控制台效率。

如果这些决策只停留在当前实现中，后续新增页面或调整现有页面时，容易出现页面结构、操作入口和文案风格漂移。因此需要把这组 UI/UX 约束沉淀为独立 ADR，作为后续修改和扩展的指导。

## 决策

1. **Web 管理面采用基础设施控制台风格**

   Web 管理面以“控制台”而不是“宣传页”为默认表达方式。页面应强调：

   - 状态可见；
   - 操作可达；
   - 结构克制；
   - 信息优先于装饰；
   - 中文优先。

   不应引入偏营销化的大 Hero、过度装饰性摘要卡片、强展示导向的大字号块或与管理任务无关的视觉噪声。

2. **页面壳层与基础结构统一**

   管理端页面应统一复用以下骨架：

   - 顶部导航壳层；
   - 页面标题区；
   - 页面状态胶囊；
   - 主操作区；
   - 数据区或结果区；
   - 抽屉 / 弹窗等局部交互容器。

   新页面优先沿用现有壳层和分区模式，不轻易引入新的页面范式。

3. **首页是控制台入口，不是产品介绍页**

   首页承担管理入口职责，应优先指向核心功能，而不是承担宣传页职责。首页可以展示：

   - 安装入口；
   - Agent 状态入口；
   - 安装包管理入口；
   - 少量能力摘要。

   首页不应为了视觉展示占用过多首屏空间。

4. **安装页采用低心智负担模型**

   安装页以“最少输入，直接执行”为原则，只暴露最终用户必须选择的参数。

   当前已接受的约束：

   - 只暴露“平台”和“发布通道”；
   - 不暴露 `control_url`；
   - 不暴露 `arch`；
   - 不暴露 `build_type`；
   - 不展示推荐版本、校验值等实现细节；
   - 由后端生成最终安装命令；
   - 命令生成时自动创建一次性注册 token；
   - 安装脚本在目标机器上检测架构，并请求对应默认安装包。

   安装页应优先展示步骤说明、必要选择项和最终命令，不应把内部部署参数或包选择细节转嫁给用户。

5. **状态页以列表为主，详情抽屉为辅**

   Agent 状态页的核心任务是检索、筛选和定位对象，因此：

   - 页面主体应保持列表优先；
   - 筛选条件应集中展示；
   - 列表列应优先服务状态判断和定位；
   - 详情信息通过侧边抽屉补充，而不是打断列表浏览。

   详情抽屉用于查看单个 Agent 的上下文，不承担完整二级页面职责。

6. **抽屉中的维护动作必须依附于版本上下文**

   Agent 详情抽屉中的维护动作不能以突兀的大主按钮单独悬挂在正文中。维护动作必须附着在“版本信息”上下文中，并遵循：

   - 先展示状态，再展示版本，再展示动作；
   - 升级 / 回滚入口使用轻量次级操作；
   - 默认视觉焦点应保持在信息本身，而非按钮；
   - 当缺少可升级版本时，应使用禁用态或说明文案，而不是保留误导性可点击入口；
   - 维护动作的具体参数编辑仍通过已有弹窗或等价轻量容器完成。

7. **中文界面优先，术语谨慎保留英文**

   Web 管理面的用户可见文案默认使用中文。只有在以下场景可以保留英文原词：

   - 命令、路径、字段名、环境变量、协议名；
   - 需要与外部标准或实际接口保持一致的术语；
   - 对用户已形成稳定认知且中文化反而降低清晰度的少数词汇。

   对 `channel`、`build type`、状态标签、按钮文案和摘要说明等可见 UI 文案，应优先提供中文表达。

8. **控制台密度优先于展示型留白**

   管理面允许保留适度留白，但首要目标是提高控制效率。默认约束：

   - 首屏尽量展示页面主要内容；
   - 除数据列表外，避免必须滚动才能看见主要操作；
   - 字体、按钮和面板内边距应控制在适中的控制台密度；
   - 列表、抽屉和弹窗应优先保证信息连贯性，而不是追求视觉夸张。

9. **状态信息不能只靠颜色表达**

   状态标签、错误、维护能力和升级状态不能只依赖颜色区分。必须同时提供文字、标签或结构性上下文，确保：

   - 低饱和界面下仍可辨识；
   - 色觉差异用户可理解；
   - 抽屉、列表和弹窗中的状态语义一致。

## 设计约束清单

后续新增或调整管理页面时，应至少自检以下问题：

- 是否仍然符合“控制台入口 / 列表优先 / 抽屉补充”的结构逻辑；
- 是否把内部实现参数错误暴露给最终用户；
- 是否为了展示感引入了不必要的大卡片、大按钮或大面积空白；
- 是否保持中文界面优先；
- 是否让主要操作出现在离相关信息最近的位置；
- 是否让状态、版本、错误和动作保持语义顺序；
- 是否复用了现有页面壳层、状态胶囊、抽屉和弹窗模式。

## 备选方案

- **不新增 ADR，只依赖现有代码实现**：实现成本最低，但后续页面扩展时很难判断哪些 UI 决策是偶然实现，哪些是长期约束，已拒绝。
- **把 UI 约束并入 ADR 0010**：可以减少文档数量，但会把能力边界、接口模型和界面交互混在一起，降低可维护性，已拒绝。
- **只写设计说明，不作为 ADR 管理**：可读性较强，但缺少“已接受决策”属性，不足以为后续实现提供稳定边界，已拒绝。

## 影响

- 后续 Web UI 修改应同时参考 ADR 0010 和本 ADR：
  - ADR 0010 负责能力边界；
  - 本 ADR 负责呈现与交互边界。
- 新增管理页面时，应优先复用既有控制台结构，而不是引入全新页面模式。
- `docs/contracts.md`、`docs/operations.md` 若新增面向用户的安装或管理说明，应与本 ADR 保持一致，避免重新暴露被刻意隐藏的参数。
- 若未来引入批量操作、任务记录、审计日志、分组管理或策略页，应继续沿用本 ADR 的结构原则，除非有新的 ADR 显式替代。

## 何时修订

出现以下情况时，应重新评估本 ADR：

- 管理面从静态页面演进为更完整的前端应用，交互模式发生显著变化；
- 用户规模和运维复杂度提升，现有控制台密度无法覆盖更复杂任务；
- 安装引导、任务编排或批量维护能力显著扩展，导致现有页面结构不再适配；
- 团队决定引入新的视觉系统或设计语言，并影响现有控制台边界。
